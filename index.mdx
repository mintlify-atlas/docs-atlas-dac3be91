# Welcome to Arnoldi

Arnoldi is a pure Python eigensolver for sparse matrices that only relies on NumPy and BLAS/LAPACK, without depending on ARPACK. It implements modern algorithms for computing eigenvalues and eigenvectors of large sparse matrices.

## What is Arnoldi?

Arnoldi is an attempt to create a viable replacement for `scipy.sparse.linalg.eigs` that removes the Fortran dependency while incorporating modern improvements in sparse eigenvalue computation. The library uses the **Krylov-Schur algorithm**, a more recent and stable approach compared to the traditional implicit restarted Arnoldi method used in ARPACK.

## Key Features

- **Pure Python Implementation**: No ARPACK or Fortran dependencies, only NumPy and BLAS/LAPACK
- **Krylov-Schur Algorithm**: Modern algorithm for large eigenvalue problems with improved stability
- **Thread-Safe & Re-entrant**: Unlike ARPACK-NG, Arnoldi is designed to be thread-safe and re-entrant
- **Multiple Orthonormalization Strategies**: Supports Modified Gram-Schmidt (MGS) and Daniel, Gragg, Kaufman, and Stewart (DGKS) methods
- **Explicit Restart with Deflation**: Efficient convergence tracking and restart capabilities
- **Built-in Test Matrices**: Includes test matrices from the sparse matrix suite plus synthetic matrices (Markov, Laplace, etc.)

## Why Arnoldi?

ARPACK-NG is a widely-used Fortran library for sparse eigenvalue solvers, but it has several limitations:

- **Not Thread-Safe**: The Fortran code is not thread-safe and, in particular, is not re-entrant, making it unsuitable for parallel applications
- **Outdated Algorithm**: It does not incorporate more recent improvements discovered for large problems, such as the Krylov-Schur algorithm introduced by G. W. Stewart in 2001 ("A Krylov-Schur Algorithm for Large Eigenproblems," SIAM J. Matrix Anal. Appl., Vol. 23, No. 3, pp. 601–614)

Arnoldi addresses these issues by providing a modern, maintainable, and thread-safe implementation in pure Python.

<Note>
  **Project Status**: Arnoldi is not yet recommended for production use. The API may change, especially in terms of convergence tracking and history. The goal is to eventually be incorporated into the SciPy codebase as a replacement for the ARPACK-based eigensolvers.
</Note>
